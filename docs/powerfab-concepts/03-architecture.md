Look at the big outer box. Everything inside it lives on one user's laptop. The two arrows leaving the bottom go out across the customer's office network. Everything outside the box is the customer's existing PowerFab infrastructure, which already works without us today.

That's the whole shape. Two processes on the laptop, two outbound connections to the customer's existing systems, no servers of ours anywhere.

## 2. The five components

### 2a. The Electron shell

The outer wrapper. The thing the user double-clicks. Its job is small but important:

- Open a window.
- Inside that window, run our React UI as if it were a web page.
- Start the C# sidecar process when the app launches.
- Stop the C# sidecar process when the app quits (so we don't leave zombie processes behind).
- Handle OS-native bits — the taskbar icon, the Start menu entry, auto-update checks, "remember window size", etc.

You don't write much code for this. Electron has a tiny "main process" entry point — about 50 to 150 lines of JavaScript — that does all of the above. The bulk of your code is the React UI it loads.

### 2b. The React/TypeScript UI

This is the existing `app/` folder in your repo. You don't have to rewrite it. It runs inside Electron's window the same way it runs inside a Chrome tab today.

The only thing that changes from today is *where the data comes from*. Right now, the UI probably reads JSON files that were pre-generated by the C# data-gen tool. In the new design, the UI makes live HTTP calls to the sidecar (e.g. `GET http://localhost:51234/api/jobs`) and gets fresh data each time.

The UI also gets a login screen — currently you might not have one, because in the old "I'll host it" model the user logged in on a website. In the new model the user types their PowerFab username and password into a form inside the Electron window. The UI passes those credentials to the sidecar, which uses them for both the SQL connection and the FabSuite API calls.

### 2c. The C# sidecar

This is the single most important new piece. It's your existing C# code (`PowerFabDataGen`) wrapped in a tiny web server.

Today, `PowerFabDataGen` is a console application. You run it from the command line and it spits out JSON files. To turn it into a sidecar, you wrap it in **ASP.NET Core minimal API** — a built-in .NET feature that turns C# functions into HTTP endpoints with very little code. Conceptually:

```csharp
// Pseudocode — about 30 lines of glue in total
var app = WebApplication.Create();

app.MapPost("/login", (LoginRequest req) => {
    // store the credentials in memory for this session
    Session.Credentials = new Credentials(req.Host, req.Port, req.User, req.Password);
    return Results.Ok();
});

app.MapGet("/api/jobs", () => {
    var client = PowerFabApiClient.Create();  // your existing code
    var jobs  = OverviewJobsModule.Fetch(client);  // your existing code
    return Results.Json(jobs);
});

app.Run("http://127.0.0.1:0");  // pick any free port
```

That's it. Your existing modules — `OverviewJobsModule`, `ProductionControlModule`, etc. — keep working exactly as they do today. You just expose them as endpoints.

The sidecar listens only on `127.0.0.1` (the same thing as `localhost`). That means it's only reachable from the same laptop. A process on another computer on the same network *cannot* connect to it. This is a security feature: if a curious coworker on the same office network pointed a browser at your sidecar's port, it would refuse the connection.

When the sidecar starts, it writes its chosen port number somewhere the Electron process can read it — usually by printing `PORT=51234` to stdout, which Electron is already listening to because it started the sidecar as a child process. Electron then knows where to send requests.

### 2d. The SQL Server connection

The PowerFab database is a Microsoft SQL Server running on the customer's network. It already exists. PowerFab itself reads and writes to it constantly.

There are two ways the dashboard can read from it, and we have a choice:

- **Option A: Through the sidecar.** The React UI asks the sidecar "give me the jobs list," and the sidecar opens a SQL connection, runs a query, returns JSON. All database access happens in C#. This is simpler because everything backend-shaped is in one language; it matches how `PowerFabDataGen` already works.
- **Option B: Directly from Electron's Node side.** The React UI asks Electron's Node process to run a query (via Electron's IPC), Node opens a SQL connection using the `mssql` package, returns JSON. The sidecar is only used for FabSuite API calls.

**Recommendation: Option A.** It keeps the architecture mentally simple ("anything that touches PowerFab goes through the sidecar"), it avoids duplicating connection-string and credential handling in two places, and it leverages all the C# code that's already written.

The cost is a tiny bit of overhead — every SQL query goes through an HTTP localhost round-trip — but for a dashboard with at most dozens of queries per minute, that's invisible.

### 2e. The FabSuite API call

We covered this in detail in `02-the-constraint.md`. The flow is:

```
React UI
   │
   │ POST http://localhost:51234/api/production-control/jobs
   ▼
C# sidecar
   │
   │ uses your existing PowerFabApiClient.cs
   │ which calls _api.ExecuteDirect(xml)
   ▼
Tekla.PowerFab.Interface.dll  (loaded in-process)
   │
   │ Tekla's mystery protocol
   ▼
FabSuite server somewhere on the network
```

The sidecar already knows how to do this part — it's literally what `PowerFabDataGen` was built for. We're just calling it from HTTP requests instead of from a command-line entry point.

## 3. Where credentials live and travel

Credentials are the most sensitive thing in this whole design, so let's be explicit.

```
   User types:  "sarah" / "hunter2"
        │
        ▼
   ┌─────────────────────────┐
   │ React UI (login screen) │
   └────────────┬────────────┘
                │ POST /login {user, password, host, port}
                ▼
   ┌─────────────────────────────────────────────────────┐
   │ C# sidecar — stores credentials in memory (RAM)     │
   │ for the lifetime of this session. Never written to  │
   │ disk. Wiped when the user logs out or quits.        │
   └──────────┬──────────────────────────┬───────────────┘
              │                          │
              │ used in connection       │ used in <Connect> block
              │ string for SQL queries   │ of every FabSuite XML
              │                          │ request
              ▼                          ▼
        SQL Server                FabSuite server
```

Important properties of this design:

- **Credentials are never sent over the public internet.** Both the SQL connection and the FabSuite request go to machines on the customer's own network.
- **Credentials are never stored on disk by our app.** Not in plain text, not encrypted, not in a "remember me" box. The user types them at every launch. (If we want a "remember me" option later, we use Windows DPAPI / the OS keychain to encrypt them. That is a small follow-up project; we don't need it for v1.)
- **Credentials never leave the laptop in any form except the two arrows above.** The React UI sends them to the sidecar once over localhost (which doesn't leave the machine). The sidecar uses them. That's it.
- **Both sides of the authentication are the customer's own systems.** The SQL Server validates the username/password against the PowerFab users table. The FabSuite server validates the same credentials against the same source. We are not running an auth service of our own. We are not introducing a new place where credentials could leak.

This is one of the strongest properties of going local. In a hosted SaaS design, we would have to store and protect credentials. Here, we never persist them at all.

## 4. What network traffic happens, in what direction, when

| When | What | Where it goes |
| --- | --- | --- |
| User launches app | nothing (sidecar starts on localhost only) | — |
| User submits login form | `POST localhost:<port>/login` | inside the laptop |
| Sidecar verifies | (a) opens SQL connection; (b) calls `fsreqPing` via DLL | customer's network |
| User navigates to a page | `GET localhost:<port>/api/<thing>` | inside the laptop |
| Sidecar fulfils request | SQL query or FabSuite call (or both) | customer's network |
| App checks for updates (every launch) | `GET <update-host>/latest.yml` | the public internet |

That's all of it. There is no traffic to our infrastructure except the update check, and even that is optional and easy to disable for air-gapped customers.

The update check deserves its own note: it's the one piece of internet connectivity the app needs. The simplest place to host it is a public bucket (S3, R2, GitHub releases). We can also make it configurable per-customer so a customer can host their own update feed inside their network if their IT department prefers.

## 5. What's already on the user's laptop vs what we have to ship

The user already has PowerFab installed (otherwise they wouldn't be buying our dashboard for it). That means these things are already there, for free:

- `C:\Program Files\Tekla\Tekla PowerFab\` — the install directory
- `Tekla.PowerFab.Interface.dll` — the FabSuite API DLL
- All the supporting Tekla DLLs that the interface DLL needs
- Network access to the FabSuite server (PowerFab itself needs this)
- Network access to the SQL Server database (PowerFab itself needs this)
- PowerFab user accounts (the same usernames/passwords we'll use)

Things our installer needs to ship:

- The Electron runtime (~80 MB — includes Chromium and Node)
- Our packaged React UI (a few MB)
- The C# sidecar `.exe`, "self-contained" so it doesn't depend on a .NET install on the user's machine (~30 MB)
- A small Windows installer wrapper around the lot
- A Start menu shortcut, an uninstaller, and an updater configuration

Total installer size: probably 100–150 MB. That's normal for Electron apps; Slack is 200+ MB, VS Code is around 100 MB.

If we ever want to shrink that, switching the shell from Electron to Tauri drops the installer to maybe 30–40 MB. The C# sidecar would stay the same size. We can revisit later.

## 6. What happens if PowerFab isn't installed

Important edge case to think about. Three scenarios:

1. **PowerFab is installed, normal location.** The sidecar starts, finds the DLL at `C:\Program Files\Tekla\Tekla PowerFab\Tekla.PowerFab.Interface.dll`, loads it, everything works.
2. **PowerFab is installed, but in a non-default location.** The sidecar can't find the DLL automatically. We show a settings screen on first launch: "Where is your PowerFab installed?" and remember the answer. The existing C# code already accepts a custom install directory (`Create(string? installDir = null)` in `PowerFabApiClient.cs`), so this is just a UI affair.
3. **PowerFab isn't installed at all.** The DLL doesn't exist anywhere. The sidecar prints a clear error: *"PowerFab does not appear to be installed on this machine. The dashboard requires PowerFab to be installed and reachable. If you believe PowerFab is installed, contact your administrator."* The UI shows that error to the user and offers a settings link.

These are all friendly failure modes that we handle in code. No mystery crashes.

## What this gives you

A design that:

- Reuses everything you've already written. React UI: stays. C# data-gen code: stays. Python scripts: stay (they're dev/analysis tools, not part of the shipped product).
- Doesn't introduce any new servers — yours or the customer's.
- Doesn't store credentials anywhere.
- Works on the network the customer already has working for PowerFab itself.
- Has a clear distribution path (Windows installer, auto-updates, same as a thousand other apps).
- Has one weird new concept to learn (the sidecar pattern) — and it's one Python is already doing in the same repo.
